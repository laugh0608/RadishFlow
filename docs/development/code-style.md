# 代码规范

更新时间：2026-05-06

## 目标

本文档用于冻结 RadishFlow 当前阶段的代码风格、抽象边界和 review 判断标准。

它不是完整语言教程，也不替代 `rustfmt`、`clippy`、C# 编译器或仓库检查脚本；它只补充那些自动化工具难以判断、但会长期影响可维护性的规则。

## 通用原则

- 代码应优先表达领域模型和稳定边界，而不是表达实现者临时的组织偏好。
- 命名必须说明真实职责、输入输出和调用意图；禁止新增含义不明的方法、类型或模块。
- 抽象必须服务明确目的：减少真实重复、隔离稳定边界、表达领域概念，或收口跨层协议。
- 不为了“未来可能扩展”预先堆叠泛化层；当前阶段的可读性和可验证性优先于空泛弹性。
- 发现根因时优先修正模型和边界，不用多层 fallback 掩盖问题。
- 新增行为应有对应测试；共享边界、错误语义、序列化契约和 interop 契约的测试优先级更高。

## 命名与抽象

- 避免只有技术包装意义的泛化名称，例如无法说明领域职责的 `helper`、`manager`、`orchestrator`、`context`、`adapter`。
- 如果确实需要使用这类词，名称中必须同时带出具体领域对象或边界，例如 `UnitOperationHostRoundOrchestrator` 比单独的 `Orchestrator` 更可接受。
- 方法名应表达可观察行为，不用 `handle`、`process`、`do_work` 这类无法说明结果的名称承载核心逻辑。
- 不把多步业务流程藏进无语义的 util 函数；流程应在调用层可读，复杂步骤再拆成有领域含义的小函数。
- 抽象层数应与稳定性匹配。仍在探索的流程不要先抽出“框架”，等边界稳定后再收口。

## Rust

- 优先使用类型系统表达约束，包括枚举、新类型、结构化错误、`Result` 和 `Option`。
- 避免用字符串常量、魔法布尔值或散落的状态标记表达核心状态机。
- 模块保持浅层职责分组；文件接近 1500 行时优先拆分同域子模块或测试 helper。
- 错误处理应保留可诊断上下文，不把不同失败折叠成一个泛化错误字符串。
- fallback 只能用于明确的兼容边界或可恢复场景；不能成为隐藏模型不一致的默认路径。
- 序列化 DTO、求解快照、文档命令和 UI presentation 应保持边界清楚，不跨层直接共享可变内部状态。

## C# / .NET

- 使用现代 .NET 风格和显式类型边界，避免恢复旧 `.NET Framework`、WinForms 或 `RegistrationServices` 时代的实现包袱。
- COM / CAPE-OPEN 语义只允许留在 `.NET` 适配层，不向 Rust Core、项目模型或通用求解接口扩散。
- interop 层应显式记录句柄、生命周期、异常映射、marshalling 形状和注册边界。
- 对外 host-facing reader、planner、runner 或 report 模型必须有稳定调用面和契约测试支撑。
- 不把 smoke test driver、PME 调试脚本或单个宿主兼容逻辑提升为通用库 API，除非边界已经被多个正式消费路径验证。

## 脚本与自动化

- PowerShell / shell 脚本应使用显式参数、清楚的 dry-run / execute 门控和可诊断输出。
- 可能写注册表、安装、反安装、打包或修改本机环境的脚本必须默认保守，并保留确认、备份和回滚边界。
- 脚本不应内置个人机器绝对路径；文档示例使用相对路径或 `<repo>` 占位。
- 自动化入口应复用仓库已有脚本和 `xtask`，避免为单次任务复制一套并行检查逻辑。

## Review Checklist

- 新增名称是否能说明领域职责，而不需要读实现才知道用途？
- 新增抽象是否减少真实重复、隔离稳定边界或表达领域概念？
- 调用方是否因为这次抽象更容易理解，而不是被迫穿过更多包装层？
- 错误语义是否足够具体，能定位阶段、对象和原因？
- 是否存在用 fallback 掩盖根因的情况？
- 测试是否覆盖了新增行为、失败路径和跨层契约？
- 文档是否只记录稳定边界和可执行规则，没有写成抽象口号？
