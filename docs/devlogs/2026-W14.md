## 关键决策

- RadishFlow 当前正式采用“本地 Rust 客户端 + 远端 `.NET 10` 控制面”的系统路线
- 远端控制面当前正式推荐采用 `ASP.NET Core / .NET 10`，不额外引入 Go 作为新的后端主语言
- 桌面端交付当前正式采用“压缩包展开后直接启动”的原生客户端形态，不追求单文件产物
- 远端控制面与资产分发属于系统外部依赖；当前仓库继续聚焦 Rust 客户端与 `.NET 10` CAPE-OPEN 适配层，不把服务端代码强行并入当前 workspace
- `rf-unitops` 第一轮统一接口当前正式收口为“标准 `MaterialStreamState` 输入输出 + 必要热力学服务注入”，不提前混入 flowsheet 调度和 FFI 细节
- `rf-flowsheet` 第一轮连接校验当前正式收口为 canonical material 端口签名、流股存在性与“一股一源一汇”，终端产品流暂允许只有 source、没有 sink
- `rf-solver` 第一轮当前正式收口为“无回路 + 内建单元 + 标准材料流股 + 最小 `SolveSnapshot`”，不提前引入 recycle、tear stream 或复杂诊断层
- Studio 当前正式收口出一条“加载物性包 -> 调用 solver -> 回写 `AppState`”的最小真实求解链路，但暂不把它误当成最终 UI 运行入口
- Studio 当前先把工作区求解入口正式收口到 `WorkspaceSolveService`，底层 `solver_bridge` 继续只负责显式 request 桥接与 provider/solver 接线
- Studio 当前已把“是否应运行”的基础判断收口到 `WorkspaceSolveService`，为后续桌面命令绑定减少 UI 侧分支判断
- Studio 当前已具备一层更接近 UI 的运行命令对象，但暂不在多包场景下隐式猜测 package，避免后续桌面行为与授权/缓存状态脱节
- Studio 当前已补出 `StudioAppFacade` 这一层显式 app command 入口，并已接到 `main.rs` 的最小 bootstrap 触发点；但最终 UI 触发点、后台任务调度和更细的运行事件流仍待继续冻结
- Studio 当前已把 `ResumeWorkspace` 与 `SetWorkspaceSimulationMode` 收口为应用命令，但真正的运行栏 / 状态栏按钮绑定口径仍未冻结

## 风险与未完成项

- 外部控制面当前只有系统设计口径，服务端代码仓库归属、部署方式和运维边界仍待后续收口
- 离线租约吊销、设备绑定和审计留存策略仍未冻结到可实现细节
- `rf-solver` 当前虽然已补出更明确的 solver-stage 失败上下文和多类端到端失败回归，但仍未覆盖 recycle、tear stream、复杂单元参数和更细的结构化诊断输出
- `rf-solver` 当前虽然已补出稳定 diagnostic code 分层与 solver-stage 失败上下文，但失败路径仍以 `RfError` 直接返回，尚未把失败诊断沉淀进 `SolveSnapshot.diagnostics`
- UI / Studio 当前虽然已能消费 `DiagnosticSummary.primary_code`，但 run panel 与更高层应用状态仍主要展示 `primary_message`，尚未正式把 stable code 纳入更细的界面分流或动作策略
- 当前 run panel 已能基于 `primary_code` 区分 solver failure notice 标题，但更细的恢复动作推荐、按钮策略和结构化失败面板仍未建立
- 当前 run panel 已能给出结构化 recovery action，但仍未把这些动作正式映射成 inspector 聚焦、选中单元或按钮级修复工作流
- 当前 run panel 已能把带 target unit 的 recovery action 映射成“选中单元 + 打开 inspector”，并补出了 widget 级 recovery 事件入口，但跨 unit 的连接修复、cycle 断开和真正的按钮级 GUI 绑定仍未收口
