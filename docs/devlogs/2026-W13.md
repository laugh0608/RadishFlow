## 关键决策

- Rust Core 与 `.NET 10` CAPE-OPEN 适配层严格解耦
- 第一阶段只导出自有 Unit Operation PMC，不支持第三方模型加载
- 当前优先级压在 `M1/M2`，不提前推进 `.NET` 复杂实现和 UI 扩展
- 开发日志采用按周分文件维护，便于阶段回顾和任务切片
- 协作规范细化为可执行规则，并与当前仓库阶段边界保持一致
- 在恢复主线功能推进前，先完成仓库治理、CI 和 App 架构这类地基工作
- dev 作为常态开发分支，master 只接受 PR，不对 dev 增加当前阶段不必要的保护
- App 架构当前采用“HYSYS 式交互 + 显式状态分层 + 独立结果快照”的折中路线
- `FlowsheetDocument` 正式收紧为可序列化真相源；文件路径、保存修订号和草稿态不再混入文档元数据
- `CommandHistory` 正式只记录语义化文档命令；运行控制和文档生命周期动作不进入撤回栈
- `SolveSessionState` 正式绑定当前观察的文档修订号，`SolvePendingReason` 只表达待办原因，不复用为错误容器
- `rf-ui` 本轮先只复用 `rf-model::Flowsheet` 和 `rf-types` 基础 ID；`DocumentId` / `SolveSnapshotId` 等 App 本地类型暂留在 UI 层，不提前上推到 Core
- RadishFlow 当前正式采用“本地求解 + 外部控制面”的混合方案：OIDC 登录、授权、派生资产分发走远端，主求解热路径继续保留在本地
- 桌面端统一视为 `public client`，后续采用系统浏览器 + PKCE + OS 安全存储，而不是内嵌长期密钥或浏览器本地存储口径
- 仓库许可采用 source-available 口径，并要求代码元数据与 `LICENSE` 文件保持一致
- 协作默认策略进一步收紧为“优先根治，不做无止境的治标兜底”
