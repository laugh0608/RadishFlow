## 关键决策

- Canvas suggestion acceptance 属于增量建模命令，不应复用完整求解前连接校验作为门槛；完整性错误继续由运行诊断和 recovery path 承担。
- 本轮只补 `Feed -> Flash Drum` 与 `Feed -> Heater/Cooler/Valve -> Flash Drum` 这类最短路径，不把该实现扩张成通用自由连线编辑器。
- `Heater / Cooler / Valve` 当前按同一类一进一出 canonical port 规则处理，但只补本地建议与 outlet 模板，不引入单元级参数面板或自由连线编辑器。
- `Mixer` 当前只在来源数量与未绑定 inlet 数量匹配时生成入口建议；多来源歧义先显式不处理，后续若要支持应引入可见的选择/确认边界。
- 真正空白项目当前只做 MVP 默认二元热力学基线初始化，不引入完整组件选择器、完整物性包选择器或项目向导；自动初始化会使空白文件打开后进入未保存状态，用户保存后才把该基线写回项目真相源。
- 显式 suggestion Apply 仍复用 `CanvasSuggestionAcceptance -> DocumentCommand::ConnectPorts`，只解决“选择哪条建议”的可控性，不引入端口点击或自由拖线。
- Canvas placement 坐标持久化只记录已提交单元的落点，并复用现有 Studio layout sidecar；它不进入 `*.rfproj.json`，不进入文档历史，也不代表已实现拖拽布局编辑器。
- 单元最新执行结果只作为 `StudioGuiWindowModel` 的只读 presentation 暴露，不回写 `FlowsheetDocument`，也不扩成结果导出、报表系统或跨会话快照历史持久化。
- 求解步骤导航继续复用 `InspectorTarget` command surface；这只是把已有结果、诊断和 Inspector 目标之间的关系前推为可点击 presentation，不改变求解器、文档模型或项目文件。
- 失败摘要的修复动作继续复用 `run_panel.recover_failure`，目标定位继续复用 `InspectorTarget` command；统一 diagnostic target action 只是 presentation 汇总，不新增另一套错误处理状态机。
- 失败详情结构化展示只消费 `DiagnosticSummary` 已有字段；port target 当前按“unit + port name”展示并定位到 unit，后续若要端口级焦点应先补正式 Inspector target 语义，而不是让 shell 私有拼 command。
- Canvas layout move 当前只作用于 Studio layout sidecar，是 GUI 布局状态，不是 `DocumentCommand::MoveUnit` 文档语义；它不触发求解 dirty，也不代表已进入自由拖拽布局编辑器。离散 nudge 的正式入口为 `canvas.move_selected_unit.left/right/up/down`，由 widget action、command registry 与 `dispatch_ui_command(command_id)` 统一派发；缺少 sidecar 坐标时允许先 pin 到 transient grid slot，避免用户必须先拖拽或重新放置单元。selection 与 command result 必须显式暴露 sidecar/transient 来源，防止该边界变成隐式 shell 行为。
- transient grid pin 仍只属于 Studio layout sidecar 初始化，不是项目几何语义迁移；即使第一次 nudge 会创建 sidecar 坐标，也必须保持项目 JSON、文档 revision、保存态和 undo/redo 历史不变。
- Canvas attention 摘要当前只属于 presentation 层：它消费 `DiagnosticSummary` / `DiagnosticSnapshot` 已经提供的 unit、stream 与 port target 结构化字段，把 port target 明确显示到 Canvas 和 object list，但 command surface 仍只定位到现有 `InspectorTarget`。
- Active Inspector 的 port attention 摘要同样只消费当前文档修订号匹配的 `DiagnosticSummary`，按当前 unit 和 `port_name` 精确匹配；它不改变 `StudioGuiInspectorTargetPortSnapshot` 的文档语义，也不引入端口级焦点或端口级编辑命令。
- Result Inspector unit-centric 视图属于 presentation 层第二个消费面，不是新的 inspector 焦点真相源：unit selection 仍是 shell 临时态，命令仍走既有 `inspector.focus_unit:*` / `inspector.focus_stream:*`；后续若要扩展更完整的结果导出、跨快照对比或单元结果历史，应继续沿正式 presentation / command 边界推进，而不是把这些语义塞进 unit-centric 折叠区。
- Result Inspector unit selection 与 stream selection 互不联动：unit-centric 视图独立显示选中单元的 unit execution result、关联步骤、关联诊断和诊断目标动作；切换流股不影响 unit selection，切换 unit 也不影响 stream selection 与对比流股，避免在一个 snapshot 下两个消费面互相覆写。

- diagnostic action 当前应被视为正式 command surface 的薄引用，不承担执行语义本身；成功路径只允许落到 `InspectorTarget`，失败路径只允许落到 `InspectorTarget` 或既有 run panel recovery，避免未来在结果区、Active Inspector、求解步骤或失败摘要中各自分叉导航/修复逻辑。
- Canvas layout 当前已收口到 sidecar placement、离散 nudge、transient grid pin 与非文档 dirty 边界；本轮验证通过后不继续扩 shell/UI 细节。后续若继续推进 Canvas layout，应先补新的正式 layout state / validation 语义，而不是直接进入拖拽布局、视口持久化或自由画布编辑器。
- 当前新增的焓值只是 MVP 常热容显热基线，参考温度固定为 `298.15 K`；它用于补齐 M2/M3 结果消费面，不代表已进入完整焓基准、相变潜热、真实 EOS/活度模型或物性包可配置参考态。
- Studio 结果区只消费求解快照中 already-materialized 的 overall/phase enthalpy，不在 shell 或 window model 里重新求热力学；缺失焓值的流股继续不显示 H summary row，避免把未求解或未闪蒸流股伪装成完整热力学结果。
- Result Inspector 的 phase comparison 只比较当前快照中已有的同名 phase 行，缺失相或缺失焓值显示为 `-`；这一步仍属于结构化结果审阅，不引入结果表格、导出或报表系统。
- Result Inspector 的 unit-centric 审阅当前只消费 solver step 中已经存在的 consumed / produced stream ids，并将它们包装成既有 `InspectorTarget` focus action；这一步只是补齐单元执行上下文，不改变求解器、项目文档、历史持久化或 Canvas 连线语义。
- Result Inspector unit-centric 的关联诊断也只消费同一组 consumed / produced stream ids，不反解析 diagnostic message，也不把入口流股诊断提升为单元编辑或自动修复语义。
- Result Inspector 和 Active Inspector 的 stream related steps 同样只使用 `StudioGuiWindowSolveStepModel` 已有的 consumed / produced stream ids；这只是让结果审阅能看到上下游执行上下文，不等同于流程图遍历、结果表格或自动布线。
- Result Inspector selector summary 只消费 `StudioGuiWindowStreamResultModel` 已经 materialized 的文本字段；`H` 行只在 `molar_enthalpy_text` 存在时出现，不在 window model 或 shell 中补算热力学。
- Result Inspector unit selector summary 同样只消费 `StudioGuiWindowSolveStepModel` 已有的 consumed / produced stream ids；输入/输出流股摘要只是结果对象选择器的只读上下文，不改变 unit selection、Inspector focus command 或项目文档语义。

## 风险与未完成项

- 当前只覆盖 source stream、`Heater / Cooler / Valve` 一进一出链路、`Mixer` 两进一出确定性链路与 Flash Drum canonical ports 的本地规则；更复杂拓扑仍需后续按同一边界补齐。
- 空白项目的默认组件与本地物性包缓存已能支撑最短建模运行闭环；但这仍是固定 MVP 二元基线，不代表已实现真实组件库、物性包浏览/切换、组分删除迁移或多物性包项目配置。
- 当前仍没有端口点击连线、拖拽布局或自动布线；显式 Apply 只是把已有本地建议从 focused-only 操作扩成可逐条选择，坐标持久化与 layout nudge 也只覆盖 sidecar 中的单元落点保存、重载和离散调整。transient grid pin 只是首次生成 sidecar 坐标的边界处理，不等同于可视拖拽或视口持久化。
- 当前结果审阅增强只补 Active Inspector 的单元执行摘要、求解步骤跳转、失败摘要动作、统一诊断目标 action 汇总、失败详情结构化展示、Canvas/object list 只读 attention 摘要、Active Inspector port attention 摘要、Result Inspector unit-centric 视图、当前快照内单元 consumed / produced stream 审阅、对应诊断归集、stream-centric 上下游 solve step 审阅，以及选择器中的已物化焓值与单元输入/输出摘要；更完整的结果表格、导出、历史对比和报表视图仍未进入本轮范围。
- 文件体积治理本轮已覆盖当前扫描范围内超过 1500 行的源码文件；后续新增实现仍需优先落在已拆出的职责模块中，避免 UI host、shell、canvas presentation、solver、`rf-ui` 与 `.NET` 适配层再次向单文件堆叠。
