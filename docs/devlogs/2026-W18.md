## 关键决策

- PME COM trace 继续保留为诊断钩子，但默认不产生文件输出；未来只有出现新的真实 PME 调用顺序或 marshalling 问题时才临时启用。
- `DWSIM / COFE` 成功路径当前作为人工 PME 回归基线，而不是自动化 PME 驱动承诺。
- `Consumes Thermodynamics`、`Supports Thermodynamics 1.0` 与 `Supports Thermodynamics 1.1` 注册分类仍只作为 material-port PME 接受条件兼容面，不改变 MVP 不实现完整 Thermo PMC 的边界。
- `typelib/RadishFlow.CapeOpen.UnitOp.Mvp.idl` 继续作为类型库真相源；`typelib/RadishFlow.CapeOpen.UnitOp.Mvp.tlb` 是从脚本生成并随项目构建/注册消费的源产物，不再靠手工命令维持。
- CAPE-OPEN / PME 主线当前进入阶段性冻结：除明确回归修复、验证脚本维护、官方接口真相源对齐和文档校准外，不再主动深挖新的 PME 边角兼容点。
- Studio 当前先补“打开示例项目 -> 运行求解 -> 查看结果/诊断”的可见闭环，不在这一轮展开大规模 UI 重排、画布编辑体验或新的求解入口；结果展示继续消费 `RunPanelWidgetModel`、`SolveSnapshot` 映射和 GUI-facing presentation DTO。
- 示例项目切换当前只覆盖仓库内置正向示例，不把 `failures/` 负向夹具当作用户入口，也不引入任意文件选择器或项目保存/另存为流程。
- 中文切换当前先限定在 `egui` shell 静态文案、运行状态和主要动作标签，不把底层 command DTO、项目文件或求解诊断真相源改成多语言模型。
- 中文字体当前采用系统字体 fallback；若未来进入跨平台打包阶段，需要再决定是否随包分发开源 CJK 字体，或继续依赖目标系统字体。
- 正式项目打开入口当前支持路径输入框和 Windows 原生打开选择器；二者都只打开现有 `*.rfproj.json`，不做保存/另存为，也不把项目打开动作写入 `*.rfproj.json`。
- 打开前确认当前只防止静默丢弃未保存改动；它不等同于保存流程，也不尝试自动写回项目文件。
- 最近项目持久化当前只冻结 shell 级 MRU 路径列表；它不持久化到项目文件，也不代表已经完成语言、布局、主题等完整应用偏好系统。
- 结构化结果展示当前仍只属于最小 Runtime presentation，不代表已经承诺完整结果 inspector、结果表格导出或长期结果审阅工作流。
- Result Inspector 当前先冻结为基于 `SolveSnapshot` 的最小 presentation 与 shell 级临时选择态；后续若继续扩完整 inspector、结果对比或导出，应优先补正式 command / state 边界，而不是在 `egui` 面板里继续堆私有展示逻辑。
- 失败运行的结果区反馈当前复用 `RunPanelNotice` 和 `WorkspaceControlState`，不改变 `SolveSessionState`；这样失败诊断、恢复动作与结果区展示保持同一个 runtime truth source。
- Inspector 焦点继续以 `rf-ui::InspectorTarget` 为真相源；Studio 只做 GUI-facing 字符串 presentation，不新建第二套选择状态。
- 诊断目标候选的交互现在固定为上下文命令而非 `egui` 私有选择态：`inspector.focus_unit:<id>` / `inspector.focus_stream:<id>` 只负责表达定位意图，目标存在性、选择集切换与 inspector 打开统一由 `rf-ui::InspectorTarget` 状态边界处理。
- 活动 Inspector 详情当前仍是只读 presentation，不承担字段编辑、保存、撤销或结果导出；后续若要做属性编辑，应继续沿字段级草稿和正式 document command 边界推进。
- Inspector 详情里的 UI 可执行项当前固定为通用 action DTO，而不是散落在具体端口、诊断或壳层控件里的临时按钮逻辑；后续新增“编辑属性”“导出结果”等动作时，也应先进入 presentation / command 边界。
