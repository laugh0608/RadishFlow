## 关键决策

- 当前先把 `M4` 的入口收口为“Rust 侧最小 FFI 可调用边界”，不在同一天同时扩张到 `.NET 10` PInvoke、COM 注册或 PME 冒烟
- `rf-ffi` 第一版先坚持“句柄 + UTF-8 + JSON + 错误码”的窄边界，避免把 Studio、auth cache 或 CAPE-OPEN 语义提前压进 Rust ABI
- 为了尽快打通调用闭环，当前 engine 先内置一份与仓库示例 flowsheet 对齐的 demo property package；真实包加载、缓存和适配层编排继续留给后续 M4 深化
- `UnitOp.Mvp` 当前把 `Ports` / `Parameters` 明确定位为“可编译、可校验的最小占位对象集合”，而不是提前承诺完整 CAPE-OPEN `Collection/Parameter/UnitPort` 运行时
- 即使继续补对象运行时，也优先先补生命周期边界、调用顺序与异常语义，不为了“像 CAPE-OPEN”而过早扩张更多接口面
- 当前先把 `UnitOp.Mvp` 的真实主线推进到“可调用 Rust 求解内核并返回最小 snapshot”，而不是先优先做注册工具或 PME 识别壳；否则只会得到“能被发现但不会算”的空 PMC
- CAPE-OPEN 实现路线正式冻结为“按官方规范独立实现 RadishFlow 自己的桥接层”，而不是复制示例代码；其中 PDF 规格书与 errata 是行为真相源，IDL/TLB/PIA 是接口形状真相源，示例代码仅作参考
- COBIA 目前只作为后续互操作选项跟踪，不进入当前阶段主线，也不因此改写既定的 COM-compatible Unit Operation PMC 路线
- 昨天未完成的“大文件拆分”当前可以结束这一轮低风险兜底治理；`src/` 下仍超过 1000 行的 `studio_app_host.rs` 与 `studio_gui_platform_host.rs` 暂不作为独立主任务继续深挖
- 后续若宿主收口任务自然触及上述重文件，可以顺手抽离局部职责；但不为了压行数而制造额外风险或伪抽象
- GUI shell 后续优先复用正式 `dispatch_ui_command(command_id)` 与 host-derived snapshot/activity 状态；对尚无 command registry 支撑的结构化交互事件，不强行包装成伪命令
- 对已经具备正式 `UiCommandModel` / `UiAction` / `StudioRuntimeTrigger` 语义的入口，优先继续删除“foreground wrapper”“typed recovery wrapper”这类一跳历史壳，不再额外扩张命令枚举
- `studio_app_host.rs` 的后续瘦身优先选择“桥接去重”和“边界收口”，而不是重新引入一层层 convenience wrapper 维持表面易用性
- `UnitOp.Mvp` 这条结果面主线当前应视为“契约已冻结、兜底已够用”的状态；后续主线重点不再是继续给同一份 report 增加更多便利读取方法，而是开始验证外部最小 host 是否只依赖这套稳定结果面就能完成状态判断和显示
- `RadishFlow.CapeOpen.SmokeTests` 的 `unitop` 路径当前明确以“最小外部 host 消费样例”作为验证目标；后续若要补宿主 helper，应建立在这条消费路径已经验证通过的前提上，而不是回到继续暴露更多内部 DTO 读取面
- 当前最小宿主 helper 继续保持为“建立在既有公开 report API 之上的复用层”，而不是回退成继续给 `RadishFlowCapeOpenUnitOperation` 增加更多 convenience accessor
- 当前最小宿主展示 helper 继续保持为“读取层 -> presentation 层”两级收口，而不是把 UI/日志展示语义继续散落在 smoke console 或未来宿主入口里各自推断
- 当前最小宿主展示 helper 继续推进为“读取层 -> presentation 层 -> formatter 层”三级收口，把 section 标题、分区顺序与 overview/detail/supplemental 的拆分规则也一并前推到库内，而不是继续让宿主入口各自拼接
- 当前“宿主驱动 PMC”这条最小 orchestration helper 先留在 `SmokeTests`，因为这一层现在的主要职责是验证调用顺序、最小输入与失败分类；在这些口径被 smoke 稳定锁定前，不急于把 driver 变成 `UnitOp.Mvp` 的正式库内 helper

## 风险与未完成项

- `rf-ffi` 当前虽然已不再是占位，但仍未覆盖 auth cache/cached package 接线、package 生命周期管理与更细粒度对象级 ABI；`.NET 10` 侧也仍只停留在薄适配与 smoke console，尚未进入 CAPE-OPEN/COM 真正实现
- 当前 `rf-ffi` 错误返回仍以状态码 + `engine_last_error_message` 为第一版口径；后续若要承接更复杂的 `.NET` 适配层诊断，需要再决定结构化错误 DTO 是否也以 JSON 形式导出
- 当前 `.NET` 解决方案级 `dotnet restore` 在这台开发机上仍会被 .NET workload SDK 解析阶段的 `MSB4276` 噪音中断；现阶段可先用逐项目 `restore/build` 作为本地验证路径，但后续仍需确认是否需要额外 solution 级兼容处理
- `studio_gui_shell` 中仍有一批直接事件入口，如窗口前后台切换、drop preview/layout mutation、native timer callback 与部分 lifecycle 事件；这些大多仍属于结构化交互事件，不适合为了“统一”而硬包装成伪 `command_id`
- `studio_app_host.rs` 与 `studio_gui_platform_host.rs` 仍是当前仅存的两个真正重文件；虽然本轮不再把它们作为独立治理任务，但后续继续叠加功能时仍需警惕职责继续膨胀
- GUI shell 仍保留部分局部交互态，如拖放会话、区域焦点与视口焦点；这些状态当前判断仍属前端会话态，后续若出现再次复制宿主事实的迹象，需要继续收口
- `studio_app_host.rs` 虽然已经连续去掉多层历史包装器，但 controller/store/state/ui-command model 仍聚在同一文件；若后续继续演进 GUI 宿主主线，仍需要结合真实职责再拆分子模块
- `.NET 10` 侧当前虽然已经补出“最小 host 消费样例”这条验证路径，但真实外部宿主仍暂时只存在于 smoke console；后续仍需继续确认这套 report 契约在更接近 PME/宿主的实际 UI 或日志组件中是否同样够用
- `UnitOperationSmokeHostDriver` 当前仍是 smoke 项目内的验证型 helper；后续若要上移到 `UnitOp.Mvp` 库内，需要先确认它抽象出来的是“真实可复用宿主编排边界”，而不是仅服务当前 console 冒烟断言的测试样板
