## 关键决策

- 当前先把 `M4` 的入口收口为“Rust 侧最小 FFI 可调用边界”，不在同一天同时扩张到 `.NET 10` PInvoke、COM 注册或 PME 冒烟
- `rf-ffi` 第一版先坚持“句柄 + UTF-8 + JSON + 错误码”的窄边界，避免把 Studio、auth cache 或 CAPE-OPEN 语义提前压进 Rust ABI
- 为了尽快打通调用闭环，当前 engine 先内置一份与仓库示例 flowsheet 对齐的 demo property package；真实包加载、缓存和适配层编排继续留给后续 M4 深化
- 昨天未完成的“大文件拆分”当前可以结束这一轮低风险兜底治理；`src/` 下仍超过 1000 行的 `studio_app_host.rs` 与 `studio_gui_platform_host.rs` 暂不作为独立主任务继续深挖
- 后续若宿主收口任务自然触及上述重文件，可以顺手抽离局部职责；但不为了压行数而制造额外风险或伪抽象
- GUI shell 后续优先复用正式 `dispatch_ui_command(command_id)` 与 host-derived snapshot/activity 状态；对尚无 command registry 支撑的结构化交互事件，不强行包装成伪命令
- 对已经具备正式 `UiCommandModel` / `UiAction` / `StudioRuntimeTrigger` 语义的入口，优先继续删除“foreground wrapper”“typed recovery wrapper”这类一跳历史壳，不再额外扩张命令枚举
- `studio_app_host.rs` 的后续瘦身优先选择“桥接去重”和“边界收口”，而不是重新引入一层层 convenience wrapper 维持表面易用性
