## 关键决策

- 本轮不把 `UnitOperationSmokeHostDriver` 整体上移为正式库 API
- 正式上移的边界是“action execution request planning”，不是“完整宿主生命周期编排”
- request planner 只消费宿主显式提供的 parameter values 与 port objects，不替宿主选择 flowsheet JSON、package id、连接对象命名或 lifecycle 调用时机
- smoke host / 未来 PME host / 其他宿主入口后续应优先复用 action plan、request planner、dispatcher 与 session snapshot，而不是各自重写 action 分类和输入缺失判断
- smoke host / 未来 PME host / 其他宿主入口后续应优先复用 action plan、request planner、host view reader、orchestrator、validation/calculation runner、host round outcome、supplemental mutation phase、follow-up 与 session snapshot，而不是各自重写 action 分类、输入缺失判断、optional mutation 注入、phase gating 和执行后下一步建议
- 未来 PME host / 其他宿主入口的最小接线应优先以 `SampleHost` 演示的正式消费路径为蓝本，而不是以 smoke driver 的验证 DSL 为蓝本
- 当前 PME-like 薄宿主入口已经证明 `SampleHost` 路径可以从“console 演示”前推为“薄宿主 session 形状”；host round 兜底子任务到这里应继续视为结束，不需要继续深挖
- 当前注册前置口径已先冻结身份和 dry-run 计划；下一步的必须补口不再是 CLSID/ProgID 选择，而是执行型 COM 注册工具的安全边界、安装/反安装路径与目标 PME 人工验证说明
- dry-run registry plan 当前用于先审查写入范围，后续若实现执行型注册，必须显式加入权限检查、路径存在性检查、备份/回滚策略和用户确认，不应直接从 plan 变成静默写注册表
- preflight 仍只允许读本机状态，不允许写 registry；真正写入必须另加显式执行门控，并单独验证回滚路径
- 目标 PME 人工验证说明已经足以结束当前“下一步如何进入真实 PME 前置准备”的文档缺口；后续必须补的是执行型注册实现和真实 PME 记录，不是继续扩 host round 兜底
- 当前子任务可正式从“host round 兜底 / 外部 host 消费路径证明”转为“M5 注册脚本固化与目标 PME 人工验证准备”；继续深挖 smoke driver 或 host round fallback 的收益已经低于推进注册入口、安装说明与 PME 记录
- 今日收尾后，M5 文档口径当前应统一视为：execute 门控、仓库脚本入口、安装/反安装运行手册与验证模板均已就位，下一步是目标 PME 选型与真实人工验证，而不是继续补注册器设计
- 对 `DWSIM / COFE` 当前兼容性问题，后续主线不再优先纠缠 `ProgID / CapeDescription / CurVer` 这类发现层注册树，而改为优先补 `TypeLib / TLB` 产物与注册链路；只要晚绑定宿主仍报 `0x80131165`，就不再把“注册成功”视为有效通过
- `Windows PowerShell 5` 默认 late binding 只能证明默认 `ICapeUtilities` 面与其返回对象的默认 interface；`ICapeUnit.Ports` 这类非默认接口应以 `QueryInterface` / 真实 PME 调用为准，不应强行要求 PowerShell 默认 binder 同时代表所有 CAPE-OPEN 接口
- 暂不推进 CO 1.1 双协议注册；本轮先按 CO 1.0 Unit Operation 的标准可选 `ICapeUnitReport` 面补齐 activation hardening，避免把 Thermo/monitoring 侧协议差异提前带入当前 PMC
- 当前判断应从继续补 CAPE-OPEN 小接口转向定位 `.NET 10 in-proc comhost` 与目标 PME 进程运行时承载冲突；trace 文件若完全不生成，则可基本判定崩在 `comhost/CoreCLR` 初始化期，后续应评估 out-of-proc COM shim 或更保守承载策略
- 由于 trace 已能记录到 constructor，当前不再是“完全未进入 managed 代码”；新增 `IPersistStreamInit` 是对 constructor 之后、automation 成员之前最可能的 PME canvas persistence 探测面做最小加固。如果下一轮 trace 仍停在 constructor，则 `.NET 10 in-proc comhost` / CoreCLR 承载冲突或其他未声明 `QueryInterface` 边界的嫌疑会进一步升高
- `DWSIM` 已验证会调用 `IPersistStreamInit.InitNew()`；后续若补入 `IPersistStorage` 后仍在 `InitNew` 返回后崩溃，优先怀疑下一个缺失面已不再是简单 persistence，而是更大的 OLE embedding 面，例如 `IOleObject`，或宿主在 `QueryInterface` 失败路径上存在 native 崩溃
- 当前新增 `IOleObject` 仍只是 canvas 添加阶段的最小 COM 兼容探测面，不代表 RadishFlow PMC 进入真实 OLE 可视嵌入、in-place activation 或 clipboard/advise 数据交换实现
- `IOleObject` 复验仍未推进 trace 调用点，因此下一轮排查必须以崩溃栈为主；在没有 dump / call stack 前，继续追加普通 COM optional interface 容易变成治标式猜测
- `dotnet-dump` 已确认 DWSIM 崩溃发生在 COM-to-CLR interface marshaling 中；当前优先做 raw pointer persistence 小修复并复验。如果修复后仍在同一阶段崩溃，再评估 out-of-proc COM shim / local server 或更保守的 in-proc shim + 外部 worker 承载策略，把 .NET 10 runtime 从 PME 进程中隔离出去
- raw pointer persistence 复验后仍在同一 marshaling 栈，说明 `IStream` 不是当前触发点；在转向 out-of-proc shim 之前，先修正更直接的 COM 形状问题：C# 接口必须与 IDL/TLB 的 `dual` 口径一致，且当前不消费的 PME host object 入参应优先以 raw pointer 接收，避免 CLR 在方法体前创建 RCW
- COFE 当前剩余问题优先按“是否仍有 PME COM material object RCW 被 UnitOp.Mvp 持有”处理；DWSIM 当前剩余问题优先按“注册分类 / OLE canvas 接受条件 / 早期 lifecycle 调用顺序”处理，不再回到 COFE simulation context optional interface 面盲补

## 风险与未完成项

- 当前 request planner 仍只覆盖 MVP action kinds，不承诺完整 CAPE-OPEN PME 生命周期编排
- 当前 orchestration / validation / calculation / host-round outcome 与 follow-up 仍只覆盖 MVP host interaction 主线，不承诺完整 PME 生命周期驱动或更高层工作流编排
- smoke driver 仍是验证型 helper，后续如果继续上移，需要先拆清真实宿主 API 与 smoke 场景 DSL 的边界
- `SampleHost` 当前虽已具备 PME-like 薄宿主 session，但仍不代表已经进入 COM 注册、PME 自动化互调或完整宿主管理框架
- `Registration` 当前虽已具备 execute 与仓库脚本入口，但验证范围仍只覆盖自有 `current-user register/unregister`；`local-machine`、PME 互调与第三方模型加载仍未进入当前阶段
- 当前已把安装/反安装说明与验证记录模板冻结到正式文档，但真实目标 PME 记录、`local-machine` 策略判断与 PME 互调本身仍未完成
- `DWSIM` 目标宿主虽已选定且注册现场已准备好，但 discovery / activation / validate / calculate 仍待人工在已启动的 GUI 中继续完成，当前新增记录先标记为 `Blocked`
- 当前 `UnitOp.Mvp` 已具备真实 `TLB / TypeLib` 产物、注册执行路径、assembly-level TLB identity 与默认 COM interface 口径；`Windows PowerShell 5` 默认 `ICapeUtilities` late-bound 探测已通过，但真实 `DWSIM / COFE` discovery / activation / validate / calculate 仍未复验
- 当前主阻塞已从 `0x80131165` 收缩到真实 PME 是否按 `ICapeUtilities / ICapeUnit / ICapeUnitReport` 等接口顺序正常消费；在跑完 `DWSIM + COFE` 前，仍不能把 M5 外部宿主互调视为闭环
- 当前执行环境里，非提权 `RegisterTypeLibForUser` 会被 OLE registry access 拒绝并触发 rollback；这条结果更像本次沙盒/权限上下文限制，不能替代用户普通桌面环境中的 HKCU 注册复验
- 当前 trace 是临时诊断代码，固定写入仓库 `artifacts/pme-trace`，不应作为长期产品行为；定位完成后需要删除或改成显式诊断开关
- 当前 `IPersistStreamInit` 只提供无状态 no-op 持久化兼容面，不代表已实现 PME 工程文件级持久化；若后续真实 PME 开始调用 `Load/Save` 并要求保留配置，需要再单独设计正式持久化契约
- 当前 `IPersistStorage` 同样只提供无状态 no-op 兼容面，不代表已实现 OLE structured storage 持久化；若 PME 后续真正要求保存配置，应先设计正式工程持久化模型，而不是把真实状态散落进这些 no-op 方法
- 当前 `IOleObject` 只提供 OLE container 探测所需的最小稳定返回，不支持真实 verb 枚举、in-place activation、clipboard data 或 advise sink；若 PME 后续真正调用这些能力，应先判断它们是否属于 CAPE-OPEN PMC 的必要兼容面，再拆出正式实现
- 当前 WER dump 配置脚本的有效采集路径会写 `HKLM\Software\Microsoft\Windows\Windows Error Reporting\LocalDumps`，必须由用户在管理员 PowerShell 中显式执行；旧的 `HKCU` 配置只能作为 legacy 清理对象，不应再作为 dump 采集路径判断依据
- 当前 DWSIM error log 中的 `AutomaticTranslation` extender `NullReferenceException` 发生在主窗口 extender 初始化阶段，时间上早于 RadishFlow 添加到画布的 trace；暂记录为宿主启动噪声候选，不作为当前 COM activation crash 主因
- 当前对 PME material object 的 `FinalReleaseComObject` 是为了避免 UnitOp.Mvp 长期持有 COFE stream RCW；若未来需要真实读取物料属性，必须重新设计短生命周期读写边界，不能把 PME material object 存入长期状态
- 新补的 thermodynamics registration categories 是 DWSIM 接受条件 probe，不代表 MVP 已实现 Thermo PMC 或第三方 property package 加载；若 DWSIM 因此推进，仍需继续确认其后续 `Ports / Parameters / Validate / Calculate` 调用顺序
