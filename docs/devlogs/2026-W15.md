## 关键决策

- `StudioAppHostController` 继续作为 app-host 级正式控制入口保留，不把 GUI 直接绑死在更底层的 `window host manager` 或 `runtime` 上
- 新增的 `StudioGuiHost` 只做 GUI-facing 适配与结果整形，不复制 app host 状态机、timer owner 规则或 command registry 逻辑
- GUI 侧后续优先通过 `dispatch_ui_command(command_id)` 与窗口生命周期入口接线，而不是重新散落 `RunPanelIntent`、foreground 推断和手工副作用读取
- GUI 生命周期事件在适配层正式收口为单一入口；“窗口前台变化”虽然底层仍走 `focus_window` 语义，但不再要求 GUI 自己区分调用路径
- native timer 也在 GUI 适配层形成正式消费面；后续真实桌面框架应优先绑定 `StudioGuiNativeTimerEffects`，而不是跨层读取 `StudioWindowTimerDriverTransition`
- GUI 侧宿主动作当前正式形成“单一命令面 + 稳定结果类型”口径；后续真实桌面框架应优先围绕 `execute_command(...)` 建立事件桥，而不是分别持有多条 method-level 调用路径
- GUI 侧现在已形成“三层桥接”口径：`StudioGuiEvent` 负责平台事件词汇，`StudioGuiHostCommand` 负责宿主命令词汇，`StudioGuiCommandRegistry` 负责菜单/命令面板消费词汇
