# 2026-W15 Development Log

周期：2026-04-06 至 2026-04-12

## 本周目标

- 把已冻结的 Studio host/controller 边界真正收口成 GUI 可直接消费的宿主适配入口
- 避免 `main.rs` 或未来真实桌面框架继续手工拼装窗口生命周期、command id 派发和宿主副作用读取

## 完成情况

- 在 `apps/radishflow-studio` 中新增 `studio_gui_host` 模块，建立 `StudioGuiHost` 这一层 GUI-facing 宿主适配器
- 为该适配器补出统一结果类型，覆盖窗口打开、窗口派发、UI command id 派发、全局事件派发和窗口关闭
- 让 GUI 侧结果直接携带最新 `ui_commands`，避免上层在命令执行后再手工回查 command registry
- 让 GUI 侧继续直接拿到既有 `runtime_report`、entitlement timer effect、native timer transition 与 ack 摘要，不重新发明第二套副作用模型
- 更新 `apps/radishflow-studio/src/main.rs`，改为直接消费 `StudioGuiHost`，不再手工串接 `StudioAppHostController`
- 继续把 GUI 侧宿主入口从“两套事件通道”收口为单一 `dispatch_lifecycle_event(...)`，统一承接：
- `WindowForegrounded { window_id }`
- `LoginCompleted`
- `NetworkRestored`
- `TimerElapsed`
- `RunPanelRecoveryRequested`
- 让 `main.rs` 的最小入口也改为通过 `StudioGuiHostLifecycleEvent` 演示生命周期派发，而不是继续分别调用 `focus_window(...)` / `dispatch_global_event(...)`
- 为新适配层补充单元测试，覆盖：
- 禁用命令时返回当前 command registry
- `run_panel.run_manual` 派发后 command registry 刷新
- 窗口关闭后的 entitlement timer retirement 摘要保持稳定
- `WindowForegrounded` 生命周期事件会正确路由到目标窗口
- `TimerElapsed` 生命周期事件在多窗口场景下会优先路由到当前 entitlement timer owner，而不是前台观察窗口

## 关键决策

- `StudioAppHostController` 继续作为 app-host 级正式控制入口保留，不把 GUI 直接绑死在更底层的 `window host manager` 或 `runtime` 上
- 新增的 `StudioGuiHost` 只做 GUI-facing 适配与结果整形，不复制 app host 状态机、timer owner 规则或 command registry 逻辑
- GUI 侧后续优先通过 `dispatch_ui_command(command_id)` 与窗口生命周期入口接线，而不是重新散落 `RunPanelIntent`、foreground 推断和手工副作用读取
- GUI 生命周期事件在适配层正式收口为单一入口；“窗口前台变化”虽然底层仍走 `focus_window` 语义，但不再要求 GUI 自己区分调用路径

## 风险与未完成项

- 当前仍是“GUI 宿主适配层已落地”，不是“真实桌面框架已完全接通”；原生窗口事件源、后台任务宿主和真正的 UI 控件绑定还未接入
- `main.rs` 目前仍是最小 bootstrap / 演示入口，尚未替换为最终的图形界面宿主
- 真实 GUI 后续仍需决定如何把窗口创建销毁、原生 timer 生命周期与后台任务入口接到 `StudioGuiHost`
