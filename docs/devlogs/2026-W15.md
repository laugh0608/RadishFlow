## 关键决策

- `StudioAppHostController` 继续作为 app-host 级正式控制入口保留，不把 GUI 直接绑死在更底层的 `window host manager` 或 `runtime` 上
- 新增的 `StudioGuiHost` 只做 GUI-facing 适配与结果整形，不复制 app host 状态机、timer owner 规则或 command registry 逻辑
- runtime 区内与 scheduler / platform / GUI activity 相关的宿主反馈，当前也应优先沉回正式 `snapshot/window model`，而不是继续停留在 `egui` 壳私有状态；后续真实桌面框架不应再复制第二套平台活动或调度说明状态机
- GUI 侧后续优先通过 `dispatch_ui_command(command_id)` 与窗口生命周期入口接线，而不是重新散落 `RunPanelIntent`、foreground 推断和手工副作用读取
- GUI 生命周期事件在适配层正式收口为单一入口；“窗口前台变化”虽然底层仍走 `focus_window` 语义，但不再要求 GUI 自己区分调用路径
- native timer 也在 GUI 适配层形成正式消费面；后续真实桌面框架应优先绑定 `StudioGuiNativeTimerEffects`，而不是跨层读取 `StudioWindowTimerDriverTransition`
- native timer 当前又进一步形成“driver runtime + platform host request”两层边界；后续真实桌面框架应优先消费 `StudioGuiPlatformTimerRequest`，而不是在入口层重复维护 due 时间差分
- 上述 platform timer request 当前也不再只是时间戳，而是正式携带 binding 身份；后续真实桌面框架可以直接据此清理/重建平台 timer，并把 callback 稳定回灌为 `window_id + handle_id`
- native timer 当前又继续形成第三层平台 adapter：真实桌面框架后续只需把各自 `native timer id` 注册/释放 API 接到 `StudioGuiPlatformTimerDriverState`，不必自己再维护逻辑 binding 影子表
- 这层 platform adapter 当前也已显式覆盖“创建失败”分支；真实桌面框架后续应在 native timer 注册失败时优先调用 failure ack，而不是把 pending 状态悬挂在适配层
- 平台 `native_timer_id` callback 当前也已显式覆盖 ignored 分支；真实桌面框架后续应优先消费 `dispatch_native_timer_elapsed_by_native_id(...)` 的 outcome，而不是把 stale/missing callback 当成异常流程
- 平台 start ack 当前也已显式覆盖 ignored + cleanup 分支；真实桌面框架后续应优先消费 `acknowledge_platform_timer_started(...)` 的 outcome 与 `follow_up_command()`，并在 stale/missing 分支立刻回收刚创建的 native timer
- 若平台 start ack 天然以批量 success/failure 回灌，真实桌面框架当前又可继续优先消费 `acknowledge_platform_timer_started_feedbacks(...)` / `acknowledge_platform_timer_start_failed_feedbacks(...)`，并在需要时直接用 `acknowledge_platform_timer_started_feedbacks_and_execute_follow_up_commands(...)` 复用批量 cleanup 顺序
- 若平台 timer API 本身是异步的，真实桌面框架当前又可继续优先消费 `dispatch_native_timer_elapsed_by_native_ids(...)` / `dispatch_due_native_timer_events_batch(...)` 这类非执行型 batch 结果，再统一执行其中 `native_timer_requests()`，不必自己拼循环与 request 抽取样板
- 若真实桌面框架希望把同一轮消息循环内的 start ack / fail ack / callback / due drain 一起提交，当前又可继续优先消费 `process_async_platform_round(...)`，复用 host 内已冻结的批处理顺序，而不是在框架层自己再拼一套 transaction glue
- 若真实桌面框架希望直接执行 host 返回的本轮平台动作，当前又可继续优先消费 `StudioGuiPlatformAsyncRound::actions()`，复用 host 内已冻结的 cleanup-before-request 顺序
- 若真实桌面框架同时具备“异步回灌 + 同步 timer 命令执行”两类平台特征，当前又可继续优先消费 `process_async_platform_round_and_execute_actions(...)`，把整轮回灌和动作执行一起压回 host glue，而不是在框架层重复保留批量 action loop
- 若平台 API 属于同步返回 `native_timer_id` 的风格，真实桌面框架后续可优先实现 `StudioGuiPlatformTimerExecutor`，并直接通过 `execute_platform_timer_request(...)` 复用 host 内已冻结的桥接顺序
- 若平台入口本来就是“收到 GUI 事件后立即调用同步 timer API”的风格，真实桌面框架当前又可继续优先消费 `dispatch_*_and_execute_platform_timer(...)` 这一层组合入口，不必在入口层重复保留 `dispatch + native_timer_request + execute` 三段样板

## 风险与未完成项

- 当前仍是“GUI 宿主适配层已落地”，不是“真实桌面框架已完全接通”；原生窗口事件源、后台任务宿主和真正的 UI 控件绑定还未接入
- `main.rs` 目前仍是最小 bootstrap / 演示入口，尚未替换为最终的图形界面宿主
- 真实 GUI 后续仍需决定如何把窗口创建销毁、原生 timer 生命周期与后台任务入口接到 `StudioGuiHost`
- 当前 native timer 已形成 GUI-facing runtime，但真实 GUI 框架的实际 timer handle 创建/释放与平台事件循环仍未正式接线
- 当前平台 timer request 已形成正式宿主适配层，但真实 GUI 仍需把各框架自己的 timer API 映射到 `Arm / Rearm / Clear`，并决定 OS/native handle 与当前 `handle_id` 的关联存放位置
- 当前平台 adapter 已形成“request -> native timer command -> callback 反查”的最小闭环，但真实 GUI 仍需把各框架自己的实际窗口句柄、消息循环和 timer 注册失败路径接进来
- 当前平台 adapter 已形成“request -> native timer command -> start success/failure ack -> callback 反查”的最小闭环，但真实 GUI 仍需把各框架自己的实际窗口句柄、消息循环和错误上报路径接进来
- 当前统一命令面已经建立，但真实 GUI 的事件循环、原生窗口句柄与平台 timer callback 仍未正式桥接到这条命令面
- 当前虽然已有第一版 `eframe/egui` GUI 宿主壳，但仍是“单原生窗口承载逻辑窗口切换”的阶段，还没有进入真正的多原生窗口宿主
- 当前 command registry 已是 GUI-facing 结构，但快捷键默认值、菜单树布局和命令面板搜索字段仍未冻结
- 当前默认绑定已经冻结到 registry，但真正的平台级快捷键冲突策略、焦点竞争规则和用户自定义覆盖仍未建立
- 当前窗口模型已经把 snapshot 整理成区域语义，但还没有进一步冻结真实桌面 GUI 所需的窗口布局状态、面板显隐与尺寸持久化契约
- 当前布局状态已接上 sidecar 持久化，并已从 `window_id` 收口到稳定 `layout_slot`；如果后续要支持更复杂窗口编排、命名工作区或显式多视图类型，仍需要继续扩展 scope 语义
- 当前 `dock_region`、`stack_group`、stack active tab、tab strip 循环切换、stack 内重排、独立 group 拆分与 region 内 placement 已进入正式 mutation，但更复杂的 dock 编排，例如 tabbed group 标题/可关闭策略、拖拽预览落点细分与跨窗口布局模板仍未建立
- 当前 `drop target` 仍是“摘要推导层 + 轻量 preview 会话态”，不包含真实 pointer hit-test、hover 生命周期状态机、拖拽占位指示条或跨窗口拖放编排
- 当前已明确的下一层边界是“GUI 可显式发 query / set preview / clear preview / apply”；这仍不等于完整 pointer drag lifecycle 状态机
- 当前 `Tab` suggestion 接受已能覆盖 material connection 的最小正式写回，但更复杂的 suggestion 形态、批量接受和更多文档命令族仍未建立
- 当前自动 suggestion 刷新只覆盖 `LocalRules`；未来若接入 `RadishMind` 并需要排序合流，还需要进一步冻结多来源 suggestion 合并策略
- 当前 runtime 区虽然已把宿主动作、平台 timer 状态和活动记录前推到正式 DTO，但 shell 内仍保留少量纯展示态辅助状态；后续继续推进真实桌面框架接线时，仍要警惕这部分再次长成第二套影子真相源
